What broke

  • OAuth/mobile auth regressions
  • CI/test instability before gates were reliable
  • UX/data assumptions that did not hold under real usage

Tradeoff I made

I chose ship and launch fast, then harden aggressively over trying to polish everything before launch.

  • Cost: more visible bugs short-term
  • Benefit: faster real-world feedback and clearer prioritization

What I learned

In the last 1-2 months, model capability has moved faster than I expected, especially with Opus 4.6 and Codex 5.3.

The main blocker for me is no longer writing code.
The blocker is judgment:

  • Picking the right direction
  • Choosing the right scope
  • Making good tradeoffs early

I’m also seeing fewer looping failure patterns where the model keeps retrying the same broken fix. With clear constraints and tests, both tools converge much more reliably.

The heaviest lift is now front-loaded:

  1. Writing a strong plan doc
  2. Tailoring Claude/Codex context docs
  3. Defining solid workflows up front

Once CI/CD is in place (GitHub Actions + automated front-end/back-end deploys after green checks), incremental shipping gets very fast.

I’m currently preferring the Codex CLI over my Claude Code + Ghostty workflow. It feels like a more modern, AI-native version of VS Code: I can stay terminal-first, still use project explorer context, and review diffs before commit/push.

I also noticed Codex picks up my custom skills without much explicit prompting. Skills like C4 Architecture, Gepetto, and Mermaid Diagrams have been especially useful while planning architecture and shaping roadmap steps.

Next focus

  • Keep tightening auth and pipeline reliability
  • Keep comparing Claude vs Codex by task category
  • Shift from velocity proxies toward product/outcome metrics
  • Integrate LangGraph into ZeroToShip’s agentic workflow for better orchestration, observability, and controlled rollout of graph-based generation

Closing thought

This is one of the most energizing times I can remember to be a builder.
